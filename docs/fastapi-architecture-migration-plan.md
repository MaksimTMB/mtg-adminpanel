# План перехода MTG AdminPanel на FastAPI и новую архитектуру

## 1) Короткий ответ на вопрос

Да, переход **реально сделать через агента и FastAPI**, но оптимальнее не «слить всё в один процесс», а собрать единый стек из:
- **Control Plane API (FastAPI)** в панели;
- **Node Agent (FastAPI)** на нодах;
- единый контракт API + единый релизный цикл для панели и агента.

Так вы получите целостную систему, где panel + agent можно менять часто, а часть MTG/MTPROTO останется изолированной и стабильной.

---

## 2) Что сейчас тормозит (по текущему репозиторию)

1. Монолитный `backend/src/app.js` с большим количеством синхронной бизнес-логики и маршрутов в одном файле.
2. SQLite + `better-sqlite3` (синхронный доступ) ограничивают масштабирование API при росте числа нод/запросов.
3. Значимая часть операций идёт через SSH fallback, что дороже по latency и нестабильно при пиках.
4. UI — большой статический `public/index.html` с инлайн-логикой; отсутствие сборочного пайплайна усложняет оптимизацию фронта.
5. Отдельные циклы поставки panel/agent уже есть, но нет единой версии контракта между ними.

---

## 3) Целевая архитектура

## 3.1 Компоненты

- **frontend**: React + Vite (замена тяжёлых Babel-цепочек и ускорение dev/build)
- **control-api**: FastAPI (REST + WebSocket для live-метрик)
- **worker**: фоновые задачи (APScheduler/Celery/RQ) для периодических проверок, автосбросов, автоудаления
- **db**: PostgreSQL (или оставить SQLite на Phase 1, затем миграция)
- **node-agent**: FastAPI на нодах (метрики контейнеров, health, версия)

## 3.2 Принципы

- Контракт panel ↔ agent версионируется (`/v1/metrics`, `agent_version`).
- SSH остаётся как временный fallback до полной установки агента на все ноды.
- Обновление panel и agent идёт из одного репозитория и одного CI/CD пайплайна.

## 4) План миграции по фазам

## Фаза 0 — Замеры и стабилизация (3–5 дней)

Цель: понять, где именно деградация.

- Добавить метрики времени по endpoint’ам (p50/p95/p99).
- Зафиксировать текущий baseline:
  - время загрузки dashboard,
  - время `/api/status`, `/api/nodes/:id/users`,
  - среднее время SSH-операций.
- Ввести request-id и структурированные логи.

Критерий выхода: есть документ с baseline и top-5 узких мест.

## Фаза 1 — Подготовка нового backend на FastAPI (1–2 недели)

- Создать `services/control-api` на FastAPI.
- Перенести базовые домены:
  - auth,
  - nodes,
  - users,
  - status/health.
- Ввести Pydantic-схемы и OpenAPI как «истину API».
- Добавить adapter слой к текущей БД, чтобы не ломать прод сразу.

Критерий выхода: FastAPI поднимается параллельно текущему Express и закрывает 30–40% read-only endpoint’ов.

## Фаза 2 — Фронтенд без тяжёлого Babel-пайплайна (1–2 недели)

- Разделить `public/index.html` на модульный фронт (React + Vite + TS).
- Вынести большие блоки в lazy chunks.
- Подключить HTTP клиент с типами от OpenAPI (генерация клиента).

Критерий выхода: dev cold start и HMR значительно быстрее, bundle меньше, UI функционально эквивалентен.

## Фаза 3 — Контракт с агентом и realtime (1 неделя)

- Нормализовать API агента: `/health`, `/metrics`, `/version`.
- Добавить совместимость версий (минимальная версия агента для панели).
- Перевести дашборд на WebSocket/SSE от control-api (агрегированные метрики), чтобы не дёргать SSH и ноды напрямую из UI.

Критерий выхода: dashboard работает через control-api + agent, SSH fallback используется редко.

## Фаза 4 — Фоновые задачи и отказ от тяжёлых sync-path (1 неделя)

- Автосброс трафика, автоудаление и периодические синки вынести в worker.
- В API оставить только быстрые запросы, тяжелое — в jobs.
- Для повторяющихся запросов ввести кэш (Redis, TTL 5–30 сек).

Критерий выхода: p95 API стабилен под типичной нагрузкой.

## Фаза 5 — Cutover и деактивация Express (2–4 дня)

- Canary: часть трафика на FastAPI.
- Сравнение ответов Express/FastAPI на критичных endpoint’ах.
- Полное переключение, freeze старого backend, затем удаление.

Критерий выхода: Express выключен, вся панель работает через FastAPI.

## 5) Как «сделать одним целым» panel + agent

Рекомендуемый вариант:

- Один mono-repo (у вас уже близко к этому).
- Единый versioning:
  - `panel_version`
  - `agent_version`
  - `api_contract_version`
- В панели реализовать:
  - проверку совместимости с агентом,
  - кнопку update agent,
  - стратегию graceful degradation при старом агенте.
- Единый CI/CD:
  - build panel image,
  - build agent image,
  - smoke tests контракта между ними.

Это даёт «единое целое» на уровне релизов, но оставляет правильную изоляцию исполнения.

## 6) Риски и как снизить

- **Риск:** миграция сразу «большим взрывом».  
  **Снижение:** Strangler pattern (параллельный FastAPI + поэтапный перенос).

- **Риск:** деградация из-за SSH путей.  
  **Снижение:** приоритетно закрыть coverage агента, SSH оставить fallback-only.

- **Риск:** несовместимость panel/agent версий.  
  **Снижение:** версионированный контракт + минимально поддерживаемая версия агента.

- **Риск:** блокировки БД/рост latency.  
  **Снижение:** переход на Postgres + async SQLAlchemy + индексы на горячие таблицы.

## 7) Практический roadmap на 30 дней

- **Неделя 1:** baseline метрик, каркас FastAPI, OpenAPI контракт.
- **Неделя 2:** перенос read endpoints + фронт на Vite skeleton.
- **Неделя 3:** realtime агрегатор + worker + Redis cache.
- **Неделя 4:** canary, полный cutover, cleanup старого Express.
